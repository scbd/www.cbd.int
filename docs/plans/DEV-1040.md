## Problem / Goal

In the **public** meeting-session view (Vue 2), when the same party re-submits a statement for the
same agenda item in a session, both uploads render as equal sibling rows. A reader cannot tell which
is current and may act on an outdated position.

**Goal:** Group a party's submissions by `meetingId + sessionId + agendaItem + partyKey`, treat the
newest (by `datetime`) as current, and mark older ones **Superseded** â€” greyed row + badge + a jump
link that scrolls to the current row. Nothing is deleted or hidden. This is **UI-derived only**: no
backend/API change, no new backend flag.

Design contract (from the ticket â€” do not redesign):
- `partyKey = government || organizationId` (GOV delegations share one `government` code; `government`
  is undefined for non-GOV orgs, so `organizationId` is the fallback).
- Within each group, sort by `datetime`; every non-newest Intervention gets `superseded: true` +
  `supersededById: <newestId>`; newest stays current.
- Additive and gated on the derived `superseded` flag. Single-submission parties and non-superseded
  rows render exactly as today.

`government`; for a non-GOV submission `government` is `undefined` and `organizationId` is present.

Contract risks to resolve here (do not assume):
- **Both `government` and `organizationId` missing** on legacy/older interventions â†’ `partyKey` would
  be falsy for several rows and they would collapse into one group and wrongly supersede each other.
  Mitigation baked into Step 1: when `partyKey` is falsy, fall back to the intervention's own `_id`
  so it forms a singleton group (never superseded). This keeps the change regression-safe.
- **`agendaItem` missing** on some rows â†’ same collapse risk within a party. Same `_id` singleton
  fallback applies when the group key is otherwise ambiguous. Confirm whether public interventions
  reliably carry `agendaItem`; if many lack it, keep the `_id` guard.
- If `government`/`organizationId`/`agendaItem` are **not** present on the public payload at all,
  **stop** â€” the feature cannot be derived client-side and the ticket's "no backend change" premise
  is invalid.

## Implementation steps

### Step 1 â€” Add the grouping/marking helper in the meetings API module

**File:** `app/components/meetings/api.js` (mirror the existing `groupInterventionBySession` /
`groupSessionByPending` helpers; add a **named export** so `view.vue` can import it alongside
`mapObjectId`).

Logic (pure function, returns a new array â€” does not mutate input objects in place beyond adding the
derived flags to shallow copies):

```
export function markSupersededInterventions(interventions) {
  // Group by meetingId + sessionId + agendaItem + partyKey.
  // partyKey = government || organizationId, else fall back to _id (singleton guard).
  const groups = new Map()
  for (const i of interventions) {
    const partyKey = i.government || i.organizationId || i._id
    const key = [i.meetingId, i.sessionId, i.agendaItem, partyKey].join('|')
    if (!groups.has(key)) groups.set(key, [])
    groups.get(key).push(i)
  }

  const supersededById = new Map() // _id -> newestId, for the older members
  for (const group of groups.values()) {
    if (group.length < 2) continue
    // newest by datetime = current
    const sorted = [...group].sort((a, b) => new Date(a.datetime) - new Date(b.datetime))
    const newest = sorted[sorted.length - 1]
    for (const i of sorted) {
      if (i !== newest) supersededById.set(i._id, newest._id)
    }
  }

  // Return copies with derived flags; untouched interventions pass through unchanged in shape.
  return interventions.map(i =>
    supersededById.has(i._id)
      ? { ...i, superseded: true, supersededById: supersededById.get(i._id) }
      : i
  )
}
```

Notes:
- Sort is **within the group only** and independent of server order, so it is correct for both the
  `datetime: 1` public list and any other order.
- Ties on identical `datetime` are unlikely for the same party+agenda; if they occur the last in
  stable-sorted order is treated as current â€” acceptable, no crash.
- Non-superseded interventions are returned unchanged (same object reference), so single-submission
  parties are byte-identical to today.

`verify:` Unit-reason by hand / console: feed a 3-element fixture (same party, same agendaItem,
ascending datetime) â†’ the two older get `superseded:true` + `supersededById` = newest `_id`, newest
has neither flag. Feed two rows with **different `organizationId` but same `government`** â†’ they group
together (GOV case). Feed a single-row party â†’ returned unchanged. Build passes (`npm run build` or the
repo's build script) with no import errors.
